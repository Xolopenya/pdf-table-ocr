# PROGRESS — распознавание буровых журналов (PDF → Excel)

Эталон: `data/raw/Дуга БЛ 4-2021.pdf`. Движок по умолчанию — **hybrid**.

## Сделано (Milestone §8)

**1. Preprocess + инвентаризация (все 22 стр.).**
`src/pipeline/preprocess.py` (приложенный модуль, используется как есть:
render_pdf → split_spread → deskew_projection → enhance → extract_grid).
`inventory.py` находит все объекты на каждой половине разворота и не теряет ни
одного: **88 объектов = 11 скважин × 8 типов** (journal_left, sampling_right,
samorodki, podschet, pokazateli, kontrol, титул, АКТ), 0 пропусков.
Дискриминатор «главная таблица vs форма» — по числу горизонталей + проверка
вертикальной решётки. Растры/overlay/JSON → `data/interim/<pdf>/`.

**2. forms_layout со ВСЕМИ спеками + detect_kind + 3 движка + key-value.**
`forms_layout.py`: 6 спек (JOURNAL_LEFT, SAMPLING_RIGHT, SAMORODKI, PODSCHET,
POKAZATELI, KONTROL); `write_spec` рисует шапку со слияниями, ряд номеров граф,
фикс-метки строк (podschet/pokazateli/kontrol), поворот вертикальных заголовков —
**структура верна by construction**. `detect_kind` по колонкам+ключевым словам.
Движки: `ocr_yandex` (handwritten/table, кэш/ретраи/dry-run), `ocr_qwen`
(OpenRouter Qwen-VL, схемный промпт), `hybrid` (**спека×строки**: колонки из
спеки пропорционально, строки из горизонталей, раскладка/ditto от Qwen, текст
ячейки от Yandex, сверка Qwen↔Yandex → needs_review). `forms_kv` — титул/АКТ.
`assemble.expand_ditto` тянет «-ll-» вниз (жёлтый); низкая уверенность/расхождение
→ оранжевый. `export.py` — листы + `_meta`.

**3. Прогон первой скважины (скв2) в hybrid live → `data/output/Дуга БЛ 4-2021__скв2.xlsx`.**
Листы: `скв2_гр1-8`, `скв2_гр9-18`, `скв2_подсчёт`, `скв2_показатели`,
`скв2_контроль`, `самородки_с1`, `титул_с1`, `акт_скв2`, `_meta`. Половины —
ОТДЕЛЬНЫМИ листами (без сшивки). Главная таблица — **10 проходок по строке**,
шапка со слияниями. Титул — 15/19 полей, АКТ — 5/7.

**4. Метрики** — `eval/score.py`: (b) число строк, (c) доля расхождений
Qwen↔Yandex (по `_meta`), (a) точность текста vs `eval/groundtruth/скв2.json`
(включится, когда появится GT).

## Фикс раскладки (детерминированная геометрия) — сделано

Раскладка больше НЕ по индексу массива движка (это давало сдвиг при пропуске
пустой графы). Теперь:
- **колонка** = X-полоса фикс-спеки (`col_fracs`, замерены по эталону), в которую
  попадает центр слова; пустая графа остаётся пустой — сдвига нет;
- **строка** = кластеризация слов Yandex по Y (рули рвутся рукописью); печатная
  шапка и подвал (подписи) отброшены; ключ строки — №/глубина;
- **текст** = покадровый OCR кропа ячейки (мелкие № и глубины — с апскейлом),
  откат на слово общего прохода, затем на значение Qwen;
- **тип содержания** (значение/пусто/«-ll-») решает Qwen; на пустых (нет чернил)
  ничего не пишем; `podschet/pokazateli/kontrol` — метки строк из спеки, движком
  не перезаписываются.

Метрики vs `eval/groundtruth/скв2.json` (strict / fuzzy текст):

| движок | итог strict | итог fuzzy | гр1-8 fuzzy | строки |
|---|---|---|---|---|
| yandex | 5.6% | 6.7% | 4/60 | 10/10 |
| qwen   | 27.0% | 28.1% | 25/60 | 10/10 |
| **hybrid** | **24.7%** | **56.2%** | **48/60 (80%)** | 10/10 |

**hybrid ≥ каждого по отдельности** (fuzzy). гр1-8 без сдвига: глубины 0,4…4,0 по
порядку, категория II/IV/…/VI, объём в своей графе; гр9-18 без сдвига: коды
пс/нз/сл в графе 11, значения в 16–18. Запуск сравнения:
`python -m eval.score out_yandex.json out_qwen.json out_hybrid.json --gt eval/groundtruth/скв2.json`.

## Стоимость разворота на 11 скважин (оценка)
~130 вызовов Yandex + ~9 Qwen на скважину → **≈1400 Yandex + ~100 Qwen** на файл.
Ориентир: Yandex Vision OCR ~$1.2/1000 → ~$1.7; Qwen3-VL-235B (OpenRouter,
даунскейл 2200px) ~$0.3–0.5. Итого **≈$2–3 за весь файл** (разово, потом кэш).

## Живой прогон скв2 + добор данных (по факту vs GT) — сделано

Живой (не dry-run) прогон скв2, затем правки по фактическим значениям. Метрика
разделена на **живые данные** (№, глубины, описание, числа опробования, значения
подсчёта) и **by-construction** (ditto/пусто/фикс-метки/единицы):

| движок | strict | fuzzy | ЖИВЫЕ fuzzy | ditto/фикс |
|---|---|---|---|---|
| yandex | 24.1% | 27.7% | 13.1% | 45.1% |
| qwen   | 53.6% | 67.0% | 54.1% | 82.4% |
| **hybrid** | **61.6%** | **76.8%** | **59.0%** | 98.0% |

Правки:
- **Выбор источника текста**: на чётком скане VLM (Qwen) читает рукопись точнее
  Yandex → в hybrid ТЕКСТ Qwen приоритетен для журнала/подсчёта; для разрежённого
  правого листа (sampling) массив Qwen ломается — там берём геометрию Yandex.
  Итог: hybrid ≥ каждого и на живых данных (59% против qwen 54%, yandex 13%).
- **Подсчёт СОШЁЛСЯ**: значения 2,4/2,0/0,4/190/76/76 на своих местах
  (пласт/масса), подпись+дата не затекают (E1:E7); из 7 значений 6 верных.
- **гр9-18**: раскладка значения по X-центру слова (не режем/не дублируем);
  калибровка граф 16/17/18 (76→16, 216→17); ditto объёма (graph 9) и коды
  пс/нз/сл нормализованы. Часть чисел (190, 48, «19 54» слиплось, «3 ик») теряет
  сам OCR — это уже не раскладка (GT их помечает low-conf).
- **score.py**: strict/fuzzy + разделение live/by-construction + merges_match +
  graph_numbers.

Осталось (OCR-уровень, не раскладка): дочитать редкие бледные числа гр9-18
(190/48/«3 ик») — кандидат на покадровый апскейл-OCR именно этих ячеек.

## GUI + наблюдаемое ядро (по одному файлу, по кнопке) — сделано

- **Ядро** `process_pdf(path, engine="hybrid", out_dir=None, save_debug=False,
  progress=...) -> {xlsx, json, report}` — единая точка входа; CLI и GUI — тонкие
  обёртки без своей логики. Один PDF → один `.xlsx` (все скважины/объекты) + `.json`.
- **Видимость**: прогресс «скважина k/N», счётчики `Yandex call/кэш/сбой`,
  `Qwen call/кэш` в статусе и логе; отчёт содержит те же счётчики.
- **Предохранители**: таймаут вызова 30–45 c, ретраи 3 с ограниченным бэкоффом;
  исчерпание → пустая ячейка + `needs_review`, без залипания; интерактивных
  ожиданий нет.
- **Устойчивость**: после КАЖДОЙ завершённой скважины досохраняется частичный
  `.xlsx/.json` — прерывание не теряет прогресс.
- **Экономия токенов**: кэш ответов по хэшу (повтор файла — из кэша); пусто/ditto
  решаются ДО сетевого вызова (OCR на них не тратится); дедуп до сборки.
- **GUI** (`run_gui.py`, Tkinter/ttk 'clam'): зона drag&drop (tkinterdnd2, иначе
  кнопка) → выбор движка (hybrid) → акцентная кнопка «Обработать» (в отдельном
  потоке, окно не виснет) → прогресс-бар + статус со счётчиками → «Открыть
  Excel/JSON/папку». Галочка «сохранять debug». Ключи из `.env`. Обработка ТОЛЬКО
  по кнопке и ТОЛЬКО по одному файлу; массовых/фоновых прогонов ядро не запускает.
  Проверено: сборка окна OK, `process_pdf` идентичен CLI (одна функция).

## Известные проблемы / следующий шаг
- **Разворот на все 11 скважин**: `process` пока обрабатывает ПЕРВУЮ скважину;
  для полного прогона нужно расширить оркестратор на все развороты — это платный
  прогон (Yandex покадрово + Qwen).
- **Коды пс/нз/сл**: Yandex читает кириллицу как латиницу (пс→nc, нз→H3) —
  занижает fuzzy на гр9-18; кандидат на нормализацию/предпочтение Qwen для кодов.
- strict у hybrid ниже fuzzy из-за мелких расхождений записи рукописи (десятичный
  разделитель уже нормализован).
- **Сдвиг колонок у Qwen**: иногда пропускает пустую графу «литологический
  разрез», сдвигая данные на 1 влево (описание попадает в графу 4 вместо 5).
  Митигировать: в промпте явно требовать слот даже для символьной графы, либо
  пост-выравнивание по широкой графе «ОПИСАНИЕ».
- **Рукописные числа**: часть глубин распознаётся с ошибками (0,8→0,3 и т.п.) —
  ожидаемо; помечены при расхождении движков.
- **podschet**: высокая доля расхождений Qwen↔Yandex (мелкая бледная таблица) —
  всё помечено оранжевым; уточнить кропы фикс-строк.
- **Регионы мини-таблиц и титул/подсчёт** берутся из фикс-долей бланка (правый
  лист расчерчен от края до края — геометрия не разделяет). Стабильно для 2021,
  проверить на 2020-х.
- GT `eval/groundtruth/скв2.json` — ждём от заказчика для метрики (a) и сравнения
  движков (hybrid ≥ каждого по отдельности).
